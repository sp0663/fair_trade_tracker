1. Abstract

The Fair Trade Supply Chain Tracker is a relational database application designed to bring transparency to the journey a product takes from the farm to the final retail shelf. Modern consumers increasingly want to verify that the food they eat or the clothes they wear have been produced under fair labour and ethical sourcing conditions, but most existing supply chains are fragmented and rely on paper based certification records that are easy to forge and difficult to audit. Our project addresses this problem by modelling the entire supply chain inside PostgreSQL, so that every stakeholder, every product batch, every ownership transfer, and every certification can be tracked, validated, and queried in a consistent way.

The system is built around five normalized tables and enforces three categories of rules directly in the database: certification validation on batch creation and transfer, immutable audit logging for every change, and a directed acyclic graph model for batch lineage that supports both splitting and merging of material through processing stages. Recursive Common Table Expressions make it possible to trace any final retail batch backward to its raw origin, or forward from any farm to every downstream product it reached. Triggers block batch creation by uncertified, expired, or revoked stakeholders before the row is ever written, which means the database itself refuses to store non-compliant data rather than depending on the application layer to do so.

The project is implemented with PostgreSQL 16, PL/pgSQL triggers and stored functions, and Python with psycopg2 and Graphviz for the visualisation and reporting layer. The result is a self-auditing database that models a realistic three chain supply network covering coffee, cocoa, and cotton products across eighteen stakeholders and twenty four tracked batches.

2. Objectives

The primary goal of this project is to design a well structured, fully normalized relational database that can track product batches through every stage of a Fair Trade supply chain and enforce ethical compliance rules automatically. The specific functional objectives of the application are listed below.

  1. End to end batch lineage. Given any batch in the system, the database must be able to return its complete ancestry back to one or more raw material origins, and also trace forward from any raw material to every downstream product that descended from it. This enables both provenance verification for consumers and rapid impact analysis during product recalls.
  2. Automated certification enforcement. Only stakeholders holding an active, non-expired, non-revoked Fair Trade certification may own or receive a batch. This rule must be enforced inside the database through triggers, not merely in application code, so that no client library or manual SQL session can bypass it.
  3. Tamper proof auditing. Every insertion of a new batch and every transfer of ownership should create an immutable record in a dedicated audit log. Once written, audit entries cannot be updated or deleted, which allows external auditors to trust the log as a faithful history.
  4. Compliance reporting. The system must expose a single view that classifies each stakeholder into one of four compliance states: COMPLIANT, EXPIRED, REVOKED, or NO CERTIFICATION. A matching stored function returns the same result for integration with reporting tools.
  5. Directed acyclic graph modelling. Real supply chains regularly split and merge material. For example, two different raw cocoa batches can be combined by a processor into a single paste batch which is then split again into multiple retail bars. The database must capture this naturally through a many to many relationship rather than forcing a tree structure.
  6. Data visualisation and reporting. A Python reporting layer should be able to connect to the live database, run every key query, and produce an HTML run report along with a rendered supply chain graph and a live Entity-Relationship diagram.

3. Scope

3.1 Boundaries

The project scope is deliberately kept to the database layer and a lightweight Python reporting shell. The following items are within scope.

  • Schema design, normalization analysis, and implementation in PostgreSQL 16.
  • Business rule enforcement through PL/pgSQL triggers and stored functions.
  • Recursive lineage queries and aggregated analytical queries.
  • Seed data that covers three realistic supply chains (coffee, cocoa, cotton) along with three deliberate edge case stakeholders that exercise every compliance state.
  • Python scripts that produce the ER diagram, the supply chain graph, and the HTML run report.

The following are explicitly outside scope and are left as reasonable extensions for future work.

  • A user facing web interface with role based login. The database models stakeholders, but there is no dedicated authentication subsystem.
  • Real time event streaming or messaging. Ownership transfers are triggered through direct function calls, not through external events.
  • Integration with external certification registries. Certification rows are captured as trusted input for this project, although in practice they would be synchronised with an external authority.
  • Horizontal scaling concerns such as sharding, read replicas, or multi region replication.

3.2 Assumptions

  • The certifications table is treated as the authoritative source of compliance information, and each stakeholder owns at most one certification row in the seed data.
  • Batch relationships form a directed acyclic graph, meaning that a batch cannot eventually become an ancestor of itself. The recursive queries therefore terminate without additional cycle detection.
  • Quantities are recorded as integers in the unit stored for each batch. Unit conversion is not performed inside the database.
  • Timestamps are recorded in the database server time zone. No explicit time zone handling beyond that is required for this project.

3.3 Limitations

  • The database currently supports one active certification per stakeholder. A design extension allowing certification history per stakeholder would require expanding the trigger logic to select the most recent active certification.
  • Ownership history for a batch is preserved only inside the audit log rather than in a dedicated ownership table, so reconstructing past ownership requires scanning audit rows.
  • The demonstration data set is intentionally small (eighteen stakeholders, twenty four batches) to keep queries readable in the report. The same schema has been verified to scale to much larger data sets without modification.

4. Schema Design and Normalization

4.1 Entities Identified

After analysing how products and certifications move through a Fair Trade supply chain, five core entities were identified.

EntityRole in the SystemPrimary Key
StakeholderAny participant in the chain: farmer, processor, distributor, or retailer.id
CertificationThe Fair Trade certification record belonging to a stakeholder, including issue date, expiry date, and active status.id
BatchA specific lot of product owned at some point by a stakeholder. Has a current owner and a quantity.id
Batch RelationA parent to child link that records the fact that a new batch was produced from one or more earlier batches. This is the associative entity for the many to many relationship between batches.(parent_batch_id, child_batch_id)
Audit LogAn append only record of every action taken on a batch (creation, update, transfer).id

4.2 Relationships Between Entities

  • Stakeholder to Certification (1 to 1 in our seed design, 1 to many in the general case). A stakeholder may hold a certification, and every certification belongs to exactly one stakeholder.
  • Stakeholder to Batch (1 to many). Each batch has exactly one current owner, and a stakeholder can own multiple batches.
  • Stakeholder to Audit Log (1 to many). Each audit entry names the actor stakeholder responsible for the event.
  • Batch to Batch (many to many, resolved through batch_relations). A child batch can be produced from multiple parent batches (merging) and a parent batch can flow into multiple child batches (splitting). This is the directed acyclic graph that gives the system its lineage capability.
  • Batch to Audit Log (1 to many). Every logged event references the batch it concerns.

4.3 Normalization Walkthrough

To justify the final schema, the design is walked through the usual normalization steps starting from a single wide unnormalized table. This is the table a beginner might sketch first: one row per stakeholder with every related piece of information packed in.

Initial unnormalized form

StakeholderInfo ( stakeholder_id, stakeholder_name, role, certifying_body, cert_issue_date, cert_expiry_date, cert_is_active, batch_ids_owned, -- repeating group: "100, 102, 105" batch_products_owned, -- repeating group: "Raw A, Proc A1, Export" parent_child_pairs, -- repeating group: "(100,102);(102,105)" audit_actions -- repeating group: "created, transferred, ..." )

This table is not even in First Normal Form. Several attributes contain comma separated lists, and a single row mixes information about a stakeholder, their certification, every batch they own, every parent to child lineage link in which they are involved, and every audit event.

First Normal Form (1NF)

1NF requires that every column holds a single atomic value and that there are no repeating groups. Each repeating group is moved into its own table, keyed against the stakeholder.

stakeholders(id, name, role, certifying_body, cert_issue_date, cert_expiry_date, cert_is_active) batches(id, product_name, quantity, unit, current_owner_id) batch_relations(parent_batch_id, child_batch_id) audit_log(id, batch_id, action, actor_id, timestamp)

The schema now satisfies 1NF. Every attribute is atomic and there are no multi valued fields.

Second Normal Form (2NF)

2NF requires 1NF plus the removal of partial dependencies on a composite primary key. The only composite key in the schema is batch_relations(parent_batch_id, child_batch_id). Since this table has no non key attributes at all, there can be no partial dependency to violate 2NF. Every other table uses a single column surrogate key, so 2NF is automatically satisfied for those as well.

Third Normal Form (3NF)

3NF requires 2NF plus the elimination of transitive dependencies, meaning no non key attribute may depend on another non key attribute. Examining stakeholders in its 1NF form exposes a violation.

Functional dependencies on stakeholders after 1NF: id -> name, role id -> certifying_body, cert_issue_date, cert_expiry_date, cert_is_active Problem: the certification attributes describe a certification, not the stakeholder. If a stakeholder is ever issued a new certification, or the certifying body renames itself, every affected row in stakeholders would have to be updated. The certification attributes are transitively dependent on the conceptual "certification identity", which is a separate entity.

The fix is to split the certification attributes into a dedicated table keyed on its own id, linked back to stakeholders through a foreign key.

stakeholders(id, name, role, created_at) certifications(id, stakeholder_id, certifying_body, issue_date, expiry_date, is_active) batches(id, product_name, quantity, unit, current_owner_id, created_at) batch_relations(parent_batch_id, child_batch_id) audit_log(id, batch_id, action, actor_id, timestamp)

After this decomposition the schema is in 3NF. Every non key attribute in every table now depends only on the primary key of that table.

Boyce-Codd Normal Form (BCNF)

BCNF strengthens 3NF by requiring that, for every non trivial functional dependency X -> Y, the left hand side X must be a superkey. Each table is checked below.

TableNon trivial FDsLHS is superkey?BCNF?
stakeholdersid → name, role, created_atYes (id is the primary key)Yes
certificationsid → stakeholder_id, certifying_body, issue_date, expiry_date, is_activeYesYes
batchesid → product_name, quantity, unit, current_owner_id, created_atYesYes
batch_relationsNone (all attributes form the key)Trivially yesYes
audit_logid → batch_id, action, actor_id, timestampYesYes

Every functional dependency in the final schema has the primary key on its left side, so the design is in BCNF. This eliminates update, insert, and delete anomalies: a certification can be revoked without touching any stakeholder row, a new batch can be added without duplicating owner information, and no lineage fact is recorded in more than one place.

4.4 Final Physical Schema

CREATE TABLE stakeholders ( id SERIAL PRIMARY KEY, name TEXT NOT NULL, role TEXT CHECK (role IN ('farmer','processor','distributor','retailer')), created_at TIMESTAMP DEFAULT NOW() ); CREATE TABLE certifications ( id SERIAL PRIMARY KEY, stakeholder_id INT REFERENCES stakeholders(id), certifying_body TEXT NOT NULL, issue_date DATE NOT NULL, expiry_date DATE NOT NULL, is_active BOOLEAN DEFAULT TRUE ); CREATE TABLE batches ( id SERIAL PRIMARY KEY, product_name TEXT NOT NULL, quantity INT NOT NULL CHECK (quantity > 0), unit TEXT DEFAULT 'kg', current_owner_id INT REFERENCES stakeholders(id), created_at TIMESTAMP DEFAULT NOW() ); CREATE TABLE batch_relations ( parent_batch_id INT REFERENCES batches(id), child_batch_id INT REFERENCES batches(id), PRIMARY KEY (parent_batch_id, child_batch_id) ); CREATE TABLE audit_log ( id SERIAL PRIMARY KEY, batch_id INT REFERENCES batches(id), action TEXT, actor_id INT REFERENCES stakeholders(id), timestamp TIMESTAMP DEFAULT NOW() );

5. Entity-Relationship Diagram

The ER diagram below was generated directly from the live PostgreSQL catalog using a Python script that introspects information_schema. This guarantees that the diagram reflects exactly what has been created in the database, rather than being a drifted hand drawn sketch.

Figure 1. Entity-Relationship diagram of the Fair Trade Supply Chain Tracker. Key icons mark primary keys and link icons mark foreign keys.

5.1 Cardinalities

  • stakeholders (1) to (0..1) certifications. A stakeholder may optionally hold a certification, and each certification belongs to exactly one stakeholder. The optionality is important because it makes it possible to model an uncertified stakeholder like our seed data id 17.
  • stakeholders (1) to (0..N) batches. A stakeholder can currently own many batches, while each batch has one current owner.
  • stakeholders (1) to (0..N) audit_log. A stakeholder can be the actor on many audit entries, while every audit entry names exactly one actor.
  • batches (M) to (N) batches, resolved through batch_relations. A batch can have many parents and many children. This many to many self relationship is what lets the schema model splits and merges inside processing stages.
  • batches (1) to (0..N) audit_log. Each batch can have many audit entries attached over its lifetime, and every audit entry belongs to exactly one batch.

5.2 Why the DAG Matters

Modelling batch_relations as a separate junction table rather than as a self referencing foreign key on batches is a deliberate design choice. A self referencing column would have forced every batch to have at most one parent, which would rule out merges. The junction table lets two parent batches (for example, Cocoa Raw A and Cocoa Raw B) be combined at a processor into a single child batch, which is a routine event in real supply chains. The trade off is that lineage queries must be expressed recursively across the junction table, which is handled by the recursive CTEs in the stored functions.

6. Application Explanation with Screenshots

This section walks through the application in action. Every figure shown below was captured from the live HTML run report produced by generate_report.py, which connects to the populated database, runs each scenario end to end, and renders the results. The figures are organised by subsystem.

6.1 Schema Overview

The first screen of the run report enumerates every object created by setup/setup.sql. The installation builds five base tables, five triggers, six views, and four non trigger stored functions. The trigger bodies (check_certification, log_batch_insert, and so on) are hidden from the function list because they are not meant to be called directly.

Figure 2. Base tables and triggers installed by the bootstrap script. Five triggers enforce certification checks on INSERT and UPDATE, log every batch event, and lock the audit log against tampering.
Figure 3. Views and non trigger stored functions. v_compliance_overview and get_compliance_report() deliberately share a result shape so they are interchangeable.

6.2 Stakeholders and Certifications

The database is populated with eighteen stakeholders spread across three supply chains (coffee, cocoa, and cotton) plus three edge case stakeholders that are essential for exercising every branch of the compliance logic: id 16 holds an expired certification, id 17 holds no certification at all, and id 18 holds a revoked certification. Keeping the certification attributes in a separate table is what makes this possible, because the stakeholder row is identical for all three edge cases and only the certification row differs.

Figure 4. All eighteen stakeholders with their roles and creation timestamps.
Figure 5. Stakeholders joined with their certification rows. The LEFT JOIN preserves id 17, whose certification fields are all NULL; the 2020 expiry on id 16 and the is_active = False on id 18 are clearly visible.

6.3 Batches and the Supply Chain DAG

Twenty four batches span the three chains. Each batch records its product name, quantity, unit, and the stakeholder currently in custody of it. The batch_relations table captures how each batch was produced from its parents, forming the directed acyclic graph that drives all lineage queries.

Figure 6. All twenty four product batches across the coffee (100 to 107), cocoa (200 to 207), and cotton (300 to 307) chains.
Figure 7. Parent to child relationships. Each row is one directed edge in the supply chain DAG; merges are visible where two different parents flow into the same child.

6.4 Recursive Lineage Queries

The core value of the database lies in its lineage queries. Given any batch, the backward trace walks through every ancestor up to the raw material origin, and the forward trace walks through every descendant to the final retail product. Both are implemented with a PostgreSQL recursive CTE that joins batch_relations back onto itself.

Figure 8. Backward lineage starting at Coffee Retail 1 (batch 106). The recursive CTE walks back through distributor, processor, and the two farm level raw batches. The depth column shows distance from the target batch.
Figure 9. Forward trace starting at Cocoa Raw A (batch 200). Every downstream product that carries material from this raw batch is returned. This is the query a quality manager would run during a product recall.

6.5 Trigger Enforcement

Four separate scenarios verify that the two certification triggers reject every non-compliant action. In each case the database returns an error and the row is never written. The application client cannot bypass these checks because they execute inside the database engine.

Figure 10. Top: attempt to insert a batch for stakeholder 17 (no certification on file). Bottom: attempt for stakeholder 16, whose certification expired on 2020-01-01. Both inserts are rejected by check_certification().
Figure 11. Top: attempt to insert a batch for stakeholder 18, whose certification has is_active = FALSE. Bottom: attempt to transfer an existing batch to uncertified stakeholder 17; the second trigger check_transfer_certification() fires on UPDATE of current_owner_id and blocks the move.

6.6 Transfer and Audit Logging

A successful transfer is performed through the transfer_batch(batch_id, new_owner_id) stored function. The function updates the ownership column, the transfer trigger validates the receiving stakeholder, and the audit trigger records the event automatically. The audit log then becomes the canonical record of who held the batch and when.

Figure 12. Batch 106 is transferred from City Coffee Retail back to Global Coffee Distributors. The before and after states are shown alongside the audit trail, which now contains a new row of action transferred written by the trigger with no application assistance.
Figure 13. Audit log immutability in action. Both UPDATE and DELETE against audit_log are rejected by prevent_audit_modification(), guaranteeing an append only history.

6.7 Views

The database exposes six views that pre-compose common queries. Three of the most important are shown below: the full compliance overview, the per chain summary, and the final products and raw materials views used for recall and provenance queries respectively.

Figure 14. v_compliance_overview classifies every stakeholder into one of four states. Thanks to the edge case seed data, all four CASE branches are reached: COMPLIANT (rows 1 to 15), EXPIRED (row 16), NO CERTIFICATION (row 17), and REVOKED (row 18).
Figure 15. v_supply_chain_summary aggregates per chain: eight batches, five stakeholders, and a total quantity that reflects the in-transit material in each of the three chains.
Figure 16. v_final_products (top) lists the leaf batches that have no downstream children, and v_raw_materials (bottom) lists the root batches that have no upstream parents. Together they bound the DAG.

6.8 Stored Functions

The three reusable lineage and compliance functions let client applications call a single routine instead of rewriting the recursive CTEs each time.

Figure 17. get_backward_lineage(207) wraps the recursive CTE used in Figure 8 into a callable function. Here it traces Chocolate Batch 2 back to both cocoa farm origins.
Figure 18. Top: get_forward_trace(300) traces Cotton Raw A forward to every downstream fabric and clothing product. Bottom: get_compliance_report() returns the same result set as v_compliance_overview, confirming that the view and the function are interchangeable.

6.9 Analytical Queries

Aggregate queries exercise GROUP BY, HAVING, and a second independent recursive CTE that computes the maximum depth of each chain from its root. These demonstrate that the schema supports realistic analytical workloads in addition to the lineage queries.

Figure 19. Total quantity per supply chain stage and the branching factor per batch. The branching factor view highlights which batches split into the largest number of children, which is useful for risk analysis.
Figure 20. Supply chain depth per root batch and a summary of audit actions. Every chain has a maximum depth of three, consistent with the farmer, processor, distributor, and retailer stages.

6.10 Supply Chain Visualisation

The final visualisation is generated by visualise/visualise_db.py, which reads the current state of the database and emits a Graphviz rendering of the entire DAG. Node colours follow stakeholder role, making the four supply chain stages visually distinct.

Figure 21. The complete supply chain graph produced from live data. Green nodes are farmer owned raw batches, blue are processor batches, orange are distributor export batches, and pink are retail ready final products. The three visually separate subgraphs are the coffee, cocoa, and cotton chains, each branching and merging through the processing stages.

6.11 Summary of Results

All functional objectives defined in Section 2 have been met. The triggers block every non-compliant insert and every non-compliant transfer; the audit log is strictly append only; the recursive functions correctly return the ancestors and descendants for every tested batch; the compliance view returns all four categorical states; and the Python reporting layer produces a complete run report together with the live ER diagram and the supply chain DAG. The schema is in BCNF, which removes update, insert, and delete anomalies from the design.
